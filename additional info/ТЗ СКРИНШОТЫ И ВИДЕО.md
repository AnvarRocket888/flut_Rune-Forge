# ТЗ: Режим скриншотов и видео-тур

## Цель

Реализовать два специальных режима запуска приложения — для создания скриншотов App Store и для записи промо-видео. Оба режима должны использовать фейковые данные, имитирующие активного пользователя, который пользуется приложением уже несколько недель.

---

## Общие требования

### Фейковые данные (Demo State)

- Изучи модели данных приложения и подготовь rich fake data, которая выглядит как реальное использование: высокий уровень профиля, заполненная коллекция объектов, прогресс в основных игровых механиках, большинство достижений разблокировано
- Данные должны быть правдоподобными: разные даты создания объектов (разброс по последним 30–60 дням), разные уровни/редкости, осмысленные имена и описания
- Demo state не должен записываться в постоянное хранилище — только в оперативную память

### Управляющие флаги

В `main.dart` должны быть два булевых константы:
```dart
const bool kScreenshotMode = false;
const bool kVideoTourMode = false;
```
Включение одного из флагов в `true` переключает приложение в соответствующий режим. В нормальном режиме оба флага `false`.

### Тема

Оба режима должны запускаться с той же темой, что и основное приложение — тёмная тема, корректные цвета навигационных панелей.

---

## Режим скриншотов (`kScreenshotMode`)

### Поведение

- Приложение запускается сразу на первом экране из заготовленной последовательности, минуя сплэш/онбординг
- Каждый экран показывается **4 секунды**, затем автоматически переходит к следующему
- Пользователь может нажать на экран, чтобы **пропустить** текущий экран вперёд
- Долгое нажатие — **начать с начала**
- Никаких наложений, индикаторов, прогресс-баров поверх экранов — только чистый UI приложения

### Последовательность экранов

- Изучи структуру навигации приложения и составь последовательность, которая охватывает все основные экраны
- Включи как минимум: главный экран, экраны коллекций/списков, экран деталей отдельного объекта, профиль пользователя, экраны с достижениями/прогрессом
- Для экрана деталей — выбери самый визуально эффектный объект из demo state (например, самый редкий или высокоуровневый)

### Реализация

Создать отдельный виджет `ScreenshotSlideshow` в `lib/screens/screenshot_slideshow.dart`. Подключить в `RuneForgeBootstrap` в `main.dart`.

---

## Режим видео-тура (`kVideoTourMode`)

### Поведение

- Приложение запускается сразу на первом экране, минуя сплэш/онбординг
- Экраны **автоматически сменяются** с плавным crossfade-переходом (≈0.5с)
- Общая длительность тура — **около 60 секунд**
- На экранах со скроллом — автоматический плавный скролл вниз через 2 секунды после появления экрана, имитируя изучение контента пользователем
- Никаких наложений, кнопок управления или индикаторов поверх экранов

### Хронометраж

Распредели время самостоятельно исходя из количества экранов, целясь в ~60 секунд суммарно. Рекомендуемый диапазон на экран: 7–10 секунд.

### Автоскролл

- Для каждого экрана-списка или длинного экрана передавай `ScrollController` и запускай `animateTo` с кривой `easeInOutSine`
- Глубина скролла — достаточная, чтобы показать 2–3 дополнительных ряда контента
- Используй `PrimaryScrollController` для передачи контроллера в дочерний виджет

### Реализация

Создать отдельный виджет `VideoTourScreen` в `lib/screens/video_tour_screen.dart`. Подключить в `RuneForgeBootstrap` в `main.dart`.

---

## Что НЕ нужно делать

- Не добавлять визуальные индикаторы касания (ripple, highlight)
- Не показывать никакие оверлеи поверх экранов в обоих режимах
- Не сохранять demo state в постоянное хранилище
- Не изменять логику и UI основного приложения
